
# Phase 1 ŌĆö AION Orchestration Diagnostics (Observe-Only)

Goal: every AION turn emits a single, queryable trace so we can see exactly what was understood, decided, written, and skipped ŌĆö with **zero** behavior change.

Already shipped (Phase 1 prelude):
- `src/diagnostics/aionTrace.ts` ŌĆö client `startTurnTrace()` mirrors stages into `aion_signals` (`kind=trace.mark`) and the diagnostics bus.
- `useAuroraChat.tsx` ŌĆö emits `turn.start | sense.dispatched | stream.start | stream.end | post.memory-writer | turn.end`.
- Feature flag `ff_aion_trace` gates client side; off by default.

What's missing vs. the brief: a dedicated trace table, cross-edge emitters (memory-writer / aion-brain / capability bus / sanitizer), and a unified panel. This plan fills exactly that, no more.

---

## 1. Data layer ŌĆö `aion_turn_traces`

New table dedicated to per-turn observation (separate from `aion_signals`, which is the analytical signal stream).

```text
aion_turn_traces
  id uuid pk
  trace_id text not null            -- generated by client tracer (`trc_ŌĆ”`)
  user_id uuid not null
  conversation_id uuid null
  route text null                   -- /index, /play, etc.
  started_at timestamptz default now()
  ended_at timestamptz null
  duration_ms int null              -- denormalized on end
  status text default 'open'        -- open | ok | error
  input_preview text null           -- first 140 chars of user msg
  intent text null
  emotion text null
  router_decision text null         -- 'reply' | 'capability' | 'followup' | 'artifact'
  capability text null
  artifact_kind text null
  language text null
  mode text null                    -- full | lite | widget
  lanes text null                   -- "live,memory,ŌĆ”"
  pillar text null
  graph_reads int default 0
  graph_writes int default 0
  signals_written int default 0
  pillar_deltas jsonb default '{}'
  brain_refreshed boolean default false
  sanitizer jsonb default '{}'      -- { matched:[], action:'clean'|'sanitized'|'rejected' }
  hebrew_cleanup jsonb default '{}'
  repetition jsonb default '{}'     -- { ok:true|false, similarity:0..1 }
  errors jsonb default '[]'
  meta jsonb default '{}'

aion_turn_trace_events
  id uuid pk
  trace_id text not null
  user_id uuid not null
  at timestamptz default now()
  source text not null              -- 'client' | 'aurora-chat' | 'memory-writer' | 'aion-brain' | 'aion-capabilities' | 'artifact-router'
  stage text not null               -- 'turn.start' | 'sense.intent' | 'sense.emotion' | 'graph.read' | 'router' | 'capability.invoked' | 'artifact.emitted' | 'stream.start' | 'stream.end' | 'sanitize' | 'memory.write' | 'graph.write' | 'signals.write' | 'pillar.delta' | 'brain.refresh' | 'turn.end' | 'error'
  data jsonb default '{}'

Indexes: (user_id, started_at desc); (trace_id); (user_id, trace_id, at) on events.
RLS: select/insert own user_id only. Service role unrestricted.
```

Both tables are observation-only. Nothing else reads them.

## 2. Server emitter ŌĆö `_shared/turnTrace.ts`

Single helper used by every edge function:

```ts
export function trace(supabase, { traceId, userId, source }) {
  return {
    event(stage, data) { /* insert into aion_turn_trace_events; never throws */ },
    upsertHeader(patch) { /* upsert into aion_turn_traces by trace_id */ },
    end(patch) { /* set ended_at, duration_ms, status */ },
  };
}
```

Rules: fire-and-forget; never blocks; service-role insert; silent on failure.

## 3. Wiring ŌĆö pass `traceId` through the pipeline

- `useAuroraChat` already mints `traceId`. Propagate it as request header `X-Aion-Trace-Id` to:
  - `aurora-chat`
  - `memory-writer`
  - `aion-capabilities`
  - `aion-brain` (when triggered from a turn)
- Each function reads the header (or generates an orphan trace if absent) and uses `_shared/turnTrace.ts`.

### Per-function events
- **aurora-chat/index.ts** ŌĆö `turn.start` (header upsert: route, mode, lanes, pillar, language, input_preview), `sense.intent`, `graph.read` (`{ nodes_pulled }`), `router` (`{ decision }`), `stream.start`, `stream.end` (`{ chars }`), `sanitize` (`{ matched, action }` from existing `detectLeaks`), `error` on any catch.
- **memory-writer/index.ts** ŌĆö `sense.emotion`, `memory.write` (`{ inserted, reinforced, skipped }`), `graph.write` (`{ nodes, edges }`), `pillar.delta` (per-pillar diff), `signals.write` (count per kind).
- **aion-capabilities/index.ts** ŌĆö `capability.invoked` (`{ capability, ok, ms }`); already mirrors a row to `aion_signals`.
- **aion-brain/index.ts** ŌĆö `brain.refresh` (`{ trigger:'turn'|'cron', mode, density, reason }`).
- **Artifact router (client)** ŌĆö when `parseArtifactSentinels` finds artifacts, emit `artifact.emitted` (`{ kind, count }`) via a new client helper that POSTs to a tiny `aion-trace-event` edge function (or inserts directly under user JWT ŌĆö simpler).

### Hebrew cleanup + repetition guard
Both already exist on the client. Hook them as new tracer marks: `tracer.mark('hebrew.cleanup', { stripped })` and `tracer.mark('repetition', { similarity, ok })`. They write into the same events table via the existing fire-and-forget path.

## 4. Closing the trace
On `turn.end` the client tracer also calls `aion-trace-close` (or inserts directly): set `ended_at`, `duration_ms`, `status`, plus aggregate counters from emitted events (`graph_writes`, `signals_written`, `pillar_deltas`, `brain_refreshed`, `sanitizer`, `repetition`).

## 6. Feature flag & rollout
- All new server emitters check `Deno.env.get('AION_TRACE') === '1'` OR presence of `X-Aion-Trace-Id` header. If neither, no-op.
- Client tracer keeps the existing `ff_aion_trace` localStorage gate.
- No code path changes when both flags off.

## Status ŌĆö Phase 2 shipped (Capability Registry binding)

- Router: new `routeCapability(intent, text)` in `_shared/capabilityRegistry.ts`. Heuristic Hebrew/English matchers map to existing capabilities.
- Wiring: `aurora-chat/index.ts` calls the router behind env flag `AION_CAPS=1`. Non-destructive caps (`progress.summarize`, `nextStep.suggest`) execute via the registry's `run()` using the same service-role client; the result is folded into the system prompt as a `[capability_result]` block.
- Destructive / side-effecting caps (`plan.restart`, `plan.delete`, `daily.generate`, `hypnosis.start`) resolve to `decision: "propose"` ŌĆö never auto-executed; the model is instructed to ask a one-line confirmation in the user's language.
- Trace: new `capability.invoked` / `capability.proposed` events on the aurora-chat trace. Header `router_decision` + `capability` set on close.
- Response headers: added `X-Aurora-Router` and `X-Aurora-Capability`, exposed via CORS.
- Off by default. Activation: `AION_CAPS=1` (edge env).
- Out of scope: client-side artifact rendering, confirmation sheet UI, and dispatching invokes from the client ŌĆö Phase 3.

## Status ŌĆö Phase 3 shipped (Client confirmation + artifact loop)

- Sentinel parser already in place (`src/lib/aion/parseArtifactSentinels.ts`); the assistant stream is scanned for `<<AION_ARTIFACT {...}>>` blocks and forwarded to `artifactBus`.
- `CapabilityInvokerBridge` (mounted in `App.tsx`) listens for `aion:capability:invoke` and dispatches to `aion-capabilities` with the propagated `traceId` so client invocations stitch into the same Phase 1 trace.
- New deterministic fallback in `useAuroraChat`: when the server router decides `propose` (`X-Aurora-Router=propose` + `X-Aurora-Capability=ŌĆ”`) and the LLM didn't emit a confirm sentinel, the client synthesizes a sticky `confirm` artifact with a labeled CTA (`plan.restart`, `plan.delete`, `daily.generate`, `hypnosis.start`). Tap ŌåÆ registry invocation ŌåÆ trace event ŌåÆ result toast.
- traceId now flows through every CTA dispatched from sentinels and from the fallback.
- Off by default end-to-end: requires `AION_CAPS=1` on the edge for the proposal headers to ever appear; the client logic is a no-op otherwise.
